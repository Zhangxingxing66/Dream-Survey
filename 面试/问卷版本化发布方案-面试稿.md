# 问卷版本化发布方案

## 1. 需求背景

这个项目原本的问卷模型只有一份数据：

- 管理端编辑页直接修改当前问卷
- 填答端读取的也是这份问卷
- 问卷发布本质上只是把 `isPublished` 改成 `true`

这会带来一个比较严重的问题：

- 一份已经在线上投放的问卷，后续如果继续在编辑器里修改标题、题目结构、选项内容，填答端会实时看到这些变化

这在真实业务里风险很高，主要体现在 4 个层面：

1. 线上稳定性差  
   已发布内容没有冻结，编辑操作会直接影响真实用户。

2. 答卷语义不一致  
   早期用户答的是旧题目，后期用户看到的是新题目，但数据都被归在同一份问卷里。

3. 统计结果失真  
   统计页如果直接用当前最新题目结构去解释历史答卷，字段和选项可能已经不匹配。

4. 缺少可追溯能力  
   无法回答“某个时间点线上到底发布的是什么版本”。

对于大厂业务来说，这种模型通常只能算 demo 级实现，无法支撑真实投放、回滚、复盘和统计分析。

---

## 2. 需求目的

这次改造的目标不是单纯“加一个版本列表”，而是把问卷从“实时编辑即线上生效”升级成“草稿编辑 + 发布快照”的发布系统。

核心目标有 5 个：

1. 编辑态和线上发布态彻底隔离  
   管理员继续改草稿，但不影响线上。

2. 填答端只消费已发布版本  
   用户永远只看到稳定快照。

3. 支持版本回滚  
   如果某次发布有问题，可以快速恢复到旧版本。

4. 保证统计解释口径稳定  
   答卷要和具体发布版本绑定，而不是只和问卷 id 绑定。

5. 为后续能力留扩展空间  
   比如版本对比、灰度发布、定时发布、按版本查看统计。

---

## 3. 技术调研与方案对比

在设计前，我先对比了三种实现路径。

### 方案一：继续使用单份问卷数据，只保留 `isPublished`

思路：

- 编辑和线上共用一份问卷记录
- 发布时只是改状态

优点：

- 实现最简单
- 改动成本最低

缺点：

- 已发布内容会被后续编辑污染
- 无法回滚
- 无法确认历史线上内容
- 统计口径不稳定

结论：

- 不满足真实业务要求，直接排除

### 方案二：发布时复制一份新问卷

思路：

- 草稿还是一份问卷
- 发布时拷贝出一份“线上问卷副本”

优点：

- 可以做到线上冻结

缺点：

- 草稿和线上副本会形成“双记录”
- 后续回滚、统计、管理都会变复杂
- 问卷列表容易出现一份业务问卷对应多条记录的问题

结论：

- 比方案一好，但模型不够自然

### 方案三：单问卷主记录 + 不可变版本快照

思路：

- 问卷主记录保存当前草稿
- 每次发布时生成一份不可变快照，写入 `versions`
- 用 `publishedVersionId` 指向当前线上版本
- 填答和统计基于发布版本读取

优点：

- 草稿和线上发布态天然分离
- 版本链路清晰
- 回滚简单
- 统计可绑定版本
- 更符合 CMS / 配置中心 / 表单平台的通用发布模型

缺点：

- 数据模型复杂度提高
- 后续接口和前端状态都要调整

最终选择：

- 采用方案三

原因：

- 它不是“为了版本而版本”，而是最符合发布系统本质的一种建模方式

## 4. 最终设计方案

### 4.1 核心设计思想

把问卷拆成两层：

- 草稿层：当前编辑中的问卷内容
- 发布层：每次发布生成的不可变版本快照

一句话概括：

`可变草稿 + 不可变快照 + 当前线上指针`

### 4.2 数据模型设计

改造后的问卷主记录包含：

```ts
{
  id,
  userId,
  title,
  desc,
  js,
  css,
  componentList,        // 当前草稿
  isDeleted,
  isStar,
  answerCount,
  createdAt,
  updatedAt,

  publishedVersionId,   // 当前线上版本
  publishedAt,
  versions: [
    {
      id,
      versionNumber,
      createdAt,
      title,
      desc,
      js,
      css,
      componentList
    }
  ]
}
```

答卷模型也做了升级：

```ts
{
  id,
  questionId,
  versionId,
  createdAt,
  answerList
}
```

关键点：

- `question.componentList` 表示当前草稿
- `question.versions[n].componentList` 表示某次发布时的快照
- `answer.versionId` 表示这份答卷是针对哪个发布版本产生的

### 4.3 为什么答卷必须绑定 `versionId`

这是这次改造里最关键的点之一。

如果答卷只记录 `questionId`，会有一个根本问题：

- 你永远无法确定答卷提交时用户实际看到的是哪一版题目

举个例子：

1. v1 问题是“你最常用的沟通方式”
2. v2 把选项改成了另一套
3. 历史答卷还保留 `phone`

如果统计页直接拿当前最新配置去解释历史答案，统计结果就会失真。

所以必须把：

- 问卷结构快照
- 用户答卷

绑定在同一版本上。

## 5. 实现细节

## 5.1 后端改造

核心文件：

- `wenjuan-server/index.js`

### 5.1.1 数据兼容与迁移

项目原来已经有旧数据，所以不能直接暴力换结构。

我在服务端读库时做了标准化处理：

- `normalizeQuestionRecord`
- `normalizeAnswerRecord`
- `normalizeDb`

作用：

1. 给旧问卷补 `versions`
2. 给旧已发布问卷自动生成一份历史发布版本
3. 给旧答卷补 `versionId`
4. 把旧结构平滑迁移到新结构

这一步的价值很大，因为它避免了“改完代码以后旧数据全失效”。

### 5.1.2 发布接口

新增接口：

- `POST /api/question/publish/:id`

流程：

1. 读取当前草稿问卷
2. 根据草稿生成快照 `createVersionSnapshotFromQuestion`
3. 版本号自增
4. 写入 `question.versions`
5. 更新 `publishedVersionId`
6. 更新 `publishedAt`

这个接口的本质不是“改状态”，而是“生成一个不可变线上版本”。

### 5.1.3 回滚接口

新增接口：

- `POST /api/question/rollback/:id`

传入：

- `versionId`

回滚逻辑：

1. 找到目标版本快照
2. 把这份快照内容覆盖回草稿
3. 同时把 `publishedVersionId` 指向该版本
4. 更新 `publishedAt`

这里我采用的是：

- 回滚到某历史版本，并让它重新成为当前线上版本

这样做的好处是：

- 草稿和当前发布态重新对齐
- 管理员回滚后可以立即继续在这版基础上编辑

### 5.1.4 公开读取接口改造

原接口：

- `GET /api/question/:id`

改造后：

- 公开访问默认返回发布版本
- 管理端带登录态且显式 `mode=draft` 时，才返回草稿

这是保证“填答端只消费发布版本”的关键约束。

换句话说：

- 不是靠前端自觉去读对的数据
- 而是后端接口层就把“公开读发布态、管理端读草稿态”固化下来

### 5.1.5 统计接口改造

原来统计只按 `questionId` 聚合。

改造后：

- `GET /api/stat/:questionId`
- `GET /api/stat/:questionId/:componentId`

都会优先基于当前 `publishedVersionId` 读取对应版本快照，并且只统计：

- `answer.questionId === currentQuestionId`
- `answer.versionId === currentPublishedVersionId`

这样可以保证：

- 当前统计看到的是当前发布版本对应的数据

### 5.1.6 答卷提交流程改造

`POST /api/answer` 现在除了 `questionId` 外，还接收：

- `versionId`

写入答卷时落库：

```ts
{
  questionId,
  versionId,
  answerList
}
```

这样统计和追溯才有依据。

## 5.2 管理端改造

核心文件：

- `wenjuan-fe/src/pages/question/Edit/EditHeader.tsx`
- `wenjuan-fe/src/hooks/useLoadQuestionData.ts`
- `wenjuan-fe/src/store/pageInfoReducer.ts`
- `wenjuan-fe/src/utils/question-store.ts`
- `wenjuan-fe/src/services/question.ts`

### 5.2.1 编辑页读取草稿

管理端编辑页加载时，显式调用：

- `getQuestionService(id, 'draft')`

这样编辑器始终操作草稿态，而不是线上发布态。

### 5.2.2 统计页读取发布版本

统计页改成：

- `useLoadQuestionData('published')`

这意味着：

- 统计页看到的是当前线上版本，而不是草稿

这非常重要，因为面试官很容易追问：

- “如果草稿已经修改但还没发布，统计页看哪一版？”

答案就是：

- 统计页看发布版，不看草稿

### 5.2.3 发布历史弹窗

编辑页新增“发布版本”入口。

能力包括：

- 拉取历史版本列表
- 标记当前发布版本
- 选择某个版本执行回滚

这部分不是为了 UI 漂亮，而是为了把“版本系统”真正暴露给用户，否则后端有能力，前端没有操作入口，面试里就讲不完整。

### 5.2.4 页面状态扩展

`pageInfoReducer` 新增了这些字段：

- `publishedVersionId`
- `publishedAt`
- `versionCount`
- `versionId`
- `versionNumber`

目的：

- 让编辑页和统计页都能知道自己当前对应的是哪个发布版本

---

## 5.3 填答端改造

核心文件：

- `wenjuan-client/src/pages/question/[id].tsx`
- `wenjuan-client/src/pages/api/answer.ts`

### 5.3.1 只读取发布版

填答端 SSR 读取问卷时，不再消费草稿。

公开访问拿到的是后端返回的发布快照，包括：

- `title`
- `desc`
- `css`
- `js`
- `componentList`
- `versionId`

### 5.3.2 提交时透传 `versionId`

填写页表单里新增隐藏字段：

```html
<input type="hidden" name="versionId" value={versionId} />
```

Next API Route 在转发答卷时，也会把 `versionId` 带给后端。

这个点很容易被忽视，但其实是整个版本化方案闭环的关键。

## 6. 过程中遇到的坑

### 6.1 旧数据兼容

这是第一坑。

原项目里的历史数据没有 `versions`、没有 `publishedVersionId`、答卷也没有 `versionId`。

如果直接切新结构，会导致：

- 已有已发布问卷无法正常读取
- 历史答卷无法参与统计

解决方式：

- 读库时做标准化和自动补全
- 把旧已发布问卷自动映射成“至少有一版发布快照”

这相当于做了一次轻量在线迁移。

### 6.2 回滚语义怎么定义

第二个坑是：

- 回滚到底只是切换线上指针，还是要同时恢复草稿？

这里其实有两种合理定义。

1. 只切线上指针  
   草稿保持不变

2. 同时恢复草稿  
   草稿和线上一起回到目标版本

我最后选了第二种。

原因：

- 对运营和产品更直观
- 回滚后可以立刻在这版基础上继续编辑
- 当前项目没有多人协同，不存在复杂冲突场景

如果以后扩展多人协同，可能会重新考虑这个设计。

### 6.3 统计是否看历史版本

第三个坑是：

- 当前统计页到底是默认看当前发布版本，还是允许看所有历史版本？

这次实现里，我先做了：

- 统计默认只看当前发布版本

原因：

- 这是最符合业务直觉的默认行为
- 接口层已经支持按 `versionId` 过滤，后续只需要补 UI 即可

这属于“底层先支持，产品层后补入口”的分步落地策略。

### 6.4 发布和保存的先后顺序

发布前，必须保证草稿已经是最新的。

所以实际流程不是直接发布，而是：

1. 先保存草稿
2. 再基于草稿生成版本快照

否则就会出现：

- 页面上看到的是最新版
- 发布出去的是旧数据

这是典型的一致性细节，如果面试官问得深，这点一定要主动提。

## 7. 底层原理

这套方案背后的核心原理有 3 个。

### 7.1 不可变快照原理

一旦发布，就把当时的问卷结构固化为一个版本快照。

优点：

- 可以精确还原历史线上内容
- 可以稳定解释历史答卷
- 可以支持版本回滚

这个思想和：

- CMS 发布版本
- 配置中心发布单
- 文档系统版本历史

本质是一致的。

### 7.2 指针切换原理

线上当前生效的不是“某份数据天然在线”，而是：

- 由 `publishedVersionId` 这个指针指向某个版本

这使得发布和回滚都变成了：

- 生成快照
- 切换指针

这种模型很稳定，也很适合后续扩展审批流、灰度流。

### 7.3 读写隔离原理

这次改造实际上做的是一种轻量级的读写隔离：

- 写操作改草稿
- 读操作分场景

具体来说：

- 管理编辑读草稿
- 公开填答读发布版
- 统计读发布版对应答卷

这和很多大厂后台系统里的“编辑态 / 线上态”分离是同一个思路。

这次改造后，项目在工程层面有几个明确提升。

### 8.1 线上稳定性更强

已发布问卷不会再被后续编辑实时污染。

### 8.2 统计口径更清晰

答卷和版本绑定，统计解释基于发布快照，更符合真实业务语义。

### 8.3 系统更可追溯

可以回答：

- 线上当前是哪版
- 某次发布是什么内容
- 历史版本能不能回退

### 8.4 面试价值更高

这个改造非常适合面试，因为它不是一个简单页面需求，而是一个完整的发布系统设计题，能覆盖：

- 数据建模
- 版本管理
- 一致性
- 前后端协同
- 兼容迁移
- 统计解释

如果让我在面试里用 1 分钟概括这次优化，我会这样说：

“这个项目原来发布后的问卷和草稿是同一份数据，导致后续编辑会直接污染线上问卷，也无法准确解释历史答卷。我把它重构成了草稿态和发布快照分离的模型：编辑器永远改草稿，发布时生成不可变快照，并用 `publishedVersionId` 指向当前线上版本；填答端只消费发布快照，答卷也绑定具体 `versionId`。这样不仅解决了线上稳定性问题，也为版本回滚、按版本统计和后续灰度发布打下了模型基础。”

## 11. 面试官可能追问的问题

### 11.1 为什么不直接用数据库事务做版本表？

当前项目后端是 JSON 文件存储，不是数据库系统，所以这次重点不是事务本身，而是先把业务模型建对。  
如果落到真实生产环境，我会把：

- `question_draft`
- `question_version`
- `answer`

拆成独立表结构，再通过事务或事件机制保证一致性。

### 11.2 为什么回滚不生成一个新版本？

这是产品策略问题，不是唯一答案。  
当前实现是“回到旧版本并重新设为当前线上版本”。  
如果业务要求更强审计能力，也可以设计成：

- 回滚操作本身再生成一个新版本

这样版本链更完整，但实现复杂度也更高。

### 11.3 为什么统计默认看当前发布版，而不是所有历史答卷？

因为对业务方来说，“当前线上版本的统计”是更自然的默认口径。  
如果把所有历史版本答卷混在一起，反而容易造成解释偏差。  
后续如果产品需要跨版本分析，可以在当前模型上继续扩展。

### 11.4 这个设计最大的 trade-off 是什么？

最大的 trade-off 是：

- 用更高的数据模型复杂度，换线上稳定性和统计一致性

也就是说：

- 简单模型容易写
- 但业务一旦上量，就会在发布、回滚、统计三个地方暴露问题

我认为这个 trade-off 是值得的。

---

## 12. 一句话总结

这次改造本质上是把问卷系统从“单记录实时生效”升级成“有草稿、有发布快照、有回滚能力的版本化发布系统”，核心价值不在 UI，而在于把数据模型和业务语义真正做对了。
