# 题型 Schema 驱动重构方案

## 1. 需求背景

这个项目本质上是一个低代码问卷平台。  
问卷编辑器的核心不是某个页面长什么样，而是“题型系统”是否足够稳定、可扩展。

原始实现里，题型能力分散在多个位置：

- 管理端有一份组件注册表
- 左侧组件库依赖这份注册表
- 右侧属性面板依赖每个题型单独写 `PropComponent`
- 统计页依赖每个题型单独写 `StatComponent`
- 填答端又有自己的一套 `if / else` 映射逻辑

这会带来几个明显问题：

1. 新增题型接入成本高  
   增加一个题型，通常要同时修改管理端组件、属性面板、填写端映射、统计渲染。

2. 题型定义不收敛  
   同一个题型的信息散落在多个文件，缺少单一可信源。

3. 可维护性差  
   题型多起来后，分发逻辑会越来越碎，回头看时很难快速理解整个系统。

4. 不利于后续低代码演进  
   如果未来要做题型插件化、动态配置、服务端下发 schema，这种分散式实现会成为阻力。

所以这个需求的本质不是“少写几个组件”，而是把题型能力从“散装实现”升级成“配置驱动的组件注册体系”。

---

## 2. 需求目的

这次重构的目标有 4 个：

1. 让题型具备统一注册入口  
   每个题型的关键元数据尽量集中在一处。

2. 让属性面板配置化  
   不再每个题型都手写一套表单，而是由 schema 驱动生成。

3. 统一管理端、填答端、统计端的题型分发逻辑  
   减少重复 if/else 和分散映射。

4. 降低新增题型成本  
   新题型主要围绕“注册 + 组件实现”展开，而不是到处找接入点。

---

## 3. 技术调研与方案对比

在设计阶段，我对比了三种思路。

### 方案一：维持现状，继续按题型各自维护

思路：

- 每个题型有自己的 `Component / PropComponent / StatComponent`
- 不做统一抽象

优点：

- 对当前代码改动最少
- 学习成本最低

缺点：

- 新增题型接入成本持续上升
- 题型定义越来越分散
- 难以演进到真正的低代码能力

结论：

- 适合 demo，不适合往工程化方向继续走

### 方案二：只做一个管理端注册表

思路：

- 管理端题型继续维持注册表
- 填答端和属性面板逻辑不动

优点：

- 改动相对可控

缺点：

- 只解决了一部分问题
- 题型能力依然没有跨端统一

结论：

- 只能算局部优化，不够彻底

### 方案三：题型元数据集中化 + Schema 驱动属性面板 + 双端 registry 分发

思路：

- 管理端维护统一题型 registry
- 每个题型声明：
  - `type`
  - `group`
  - `defaultProps`
  - `propSchema`
  - `Component`
  - `StatComponent`
- 右侧属性面板通过 `propSchema` 动态生成
- 填答端改造成基于 registry 的映射分发

优点：

- 题型定义更收敛
- 属性面板可配置化
- 新题型接入路径清晰
- 更接近真正的低代码模型

缺点：

- 重构范围更大
- 类型设计和运行时配置之间需要平衡

最终选择：

- 采用方案三

因为这个项目的核心竞争力就在题型系统，题型不抽象好，后续发布、统计、版本化都会被拖累。

---

## 4. 最终设计方案

## 4.1 核心思想

把题型定义从“多个页面各自关心一点”收敛成“题型 registry 统一描述”。

一句话概括：

`题型 = 元数据 + 默认配置 + 编辑器渲染 + 属性 Schema + 统计适配`

### 4.2 管理端 registry 设计

管理端题型统一定义在：

- `wenjuan-fe/src/components/QuestionComponents/index.ts`

每个题型配置包含：

```ts
{
  title,
  type,
  groupId,
  groupName,
  Component,
  defaultProps,
  propSchema,
  StatComponent?
}
```

这里最关键的是 `propSchema`。

因为过去属性面板是：

- 每个题型一个 `PropComponent`

现在变成：

- 每个题型提供一份字段 schema
- 通用表单渲染器根据 schema 生成属性编辑 UI

### 4.3 通用 SchemaForm 设计

我新增了：

- `wenjuan-fe/src/components/QuestionComponents/SchemaForm.tsx`

它负责根据字段 schema 渲染通用属性表单，目前支持：

- `input`
- `textarea`
- `select`
- `checkbox`
- `radioOptions`
- `checkboxOptions`

这样像这些题型：

- 输入框
- 多行输入
- 标题
- 段落
- 问卷信息

它们的属性面板都不需要再写专门的表单组件了，只要配 schema 就能生成。

### 4.4 属性面板重构

原先：

- `ComponentProp.tsx` 根据题型拿 `PropComponent`

现在：

- `ComponentProp.tsx` 根据题型拿 `propSchema`
- 再通过 `SchemaForm` 渲染统一属性面板

这样右侧面板的职责发生了变化：

- 从“渲染具体题型表单组件”
- 变成“渲染题型 schema”

这是这次重构里最核心的变化之一。

### 4.5 组件库分组自动生成

过去组件库分组依赖 `componentConfGroup` 手工维护。

现在改成：

- registry 中每个题型自带 `groupId / groupName`
- 系统自动聚合成分组列表

这意味着新增题型时：

- 不需要再手改一份额外的 group 配置

### 4.6 填答端 registry 化

填写端原先是：

```ts
if (type === 'questionInput') ...
if (type === 'questionRadio') ...
```

这种写法的问题不是“代码长”，而是它把题型映射关系散落在逻辑判断里。

现在改造成：

- `wenjuan-client/src/components/QuestionComponents/index.tsx`

维护一份 client registry：

```ts
[
  { type, render }
]
```

然后统一通过：

- `getClientComponentByType(type)`

分发渲染。

虽然填写端还没有完全复用管理端 registry，但已经从硬编码分支升级成了注册式分发，为后续共享 schema 打下基础。

### 4.7 统计适配逻辑保留在 registry

对于可统计题型，比如：

- `questionRadio`
- `questionCheckbox`

registry 中保留：

- `StatComponent`

这样统计页就能通过题型定义直接拿到对应的图表渲染器，而不是在页面层自己维护映射。

## 5. 实现细节

## 5.1 为什么没有强行做“前后端完全共享 schema”

面试官很可能会问：

- “既然你都做 Schema 驱动了，为什么不把管理端和填写端完全共用一套 schema？”

答案是：

- 我做的是分阶段演进，而不是一步到位重构

当前项目的技术现实是：

- 管理端和填写端是两个独立项目
- 组件实现不完全一致
- 填写端是面向真实表单提交的 HTML/Next 页面
- 管理端是 Ant Design 驱动的编辑器预览

所以这次我先统一的是：

1. 管理端题型元数据
2. 管理端属性配置方式
3. 填写端分发模式

而不是硬把两个端的所有实现揉成一个共享包。

这是一个典型的工程 trade-off：

- 优先收敛核心抽象
- 再逐步推进更深层的共享

## 5.2 SchemaForm 如何处理动态选项

像单选、多选这种题型，属性面板不是静态字段，而是：

- 可增删选项
- 选项值需要自动补齐
- 默认选中值要跟选项联动

所以我在 `SchemaForm` 里补了两类特殊字段：

- `radioOptions`
- `checkboxOptions`

并在内部做了几件事：

1. 自动过滤空选项
2. 自动补齐 `value`
3. 当默认值不在选项内时清空
4. 支持动态表单数组增删

这一步很关键，因为如果 schema 只能描述静态字段，那它对复杂题型的覆盖能力就不够。

## 5.3 为什么保留原有 `Component` 和 `StatComponent`

这次我重构的是“题型配置体系”，不是“所有组件全部重写”。

所以我保留了：

- 画布渲染组件 `Component`
- 统计图组件 `StatComponent`

而把变化重点放在：

- 元数据声明
- 属性面板渲染
- 分发方式

原因是这样能把收益集中在系统抽象层，同时控制重构风险。

## 6. 遇到的坑

### 6.1 类型系统和运行时 schema 的平衡

这是最核心的坑。

TypeScript 擅长描述静态类型，但 `propSchema` 是运行时配置。  
也就是说：

- 组件 props 是编译期约束
- schema 是运行期解释

如果强行追求绝对严格的类型推导，代码复杂度会很高。  
所以这次我做的平衡是：

- 管理端 registry 统一用 `Record<string, any>` 作为 props 抽象
- 把重点放在接入结构统一，而不是极限类型体操

这是工程实践里比较现实的选择。

### 6.2 填写端组件 props 定义更严格

填写端各组件的 props 声明比较具体，而 registry 传入的是统一的 `props` 容器。  
重构过程中出现了类型收窄冲突。

我的处理方式是：

- 保持填写端 registry 的统一分发
- 在渲染层做显式类型放宽

因为填写端这层的目标是统一接入路径，而不是在 registry 层再做一遍强类型建模。

### 6.3 不能为了重构而重构

还有一个很实际的问题：

- 是否要把所有旧的 `PropComponent` 全删掉？

我没有这么做。

因为当前项目里这些旧组件本身不是 bug，它们只是旧接入方式。  
完全删除会扩大改动范围，也会增加回滚成本。

所以这次是：

- 新逻辑接管主路径
- 旧实现保留作为历史资产

这更符合真实团队里的渐进式重构策略。

---

## 7. 底层原理

这次重构背后的原理，可以归纳成 3 个词：

### 7.1 单一可信源

以前题型定义散落在多个地方。  
现在 registry 成为题型元数据的主要入口。

这就是典型的：

- Single Source of Truth

### 7.2 配置驱动 UI

属性面板不再由每个题型自己写死，而是：

- `schema -> 表单 UI`

这是一种典型的配置驱动设计。  
它的价值不在于少写代码，而在于让系统具备规模化扩展能力。

### 7.3 注册表模式

无论管理端还是填写端，本质上都从“条件分发”走向了“注册表分发”。

也就是说：

- 新能力通过注册接入
- 而不是修改一长串页面判断逻辑

这是一种典型的可扩展架构模式。

---

## 8. 收益

这次重构带来的收益主要有 4 个。

### 8.1 新题型接入成本下降

现在新增题型时，开发者主要关注：

1. 题型渲染组件
2. 默认配置
3. 属性 schema
4. 是否有统计渲染

接入路径比以前清晰很多。

### 8.2 属性面板统一化

过去每个题型都在重复写表单胶水代码。  
现在常见字段都能通过 `SchemaForm` 复用。

### 8.3 代码结构更适合继续演进

后续如果要做：

- 服务端下发题型 schema
- 题型插件化
- 题型市场
- 组件配置导出

现在的架构比之前更容易承接。

### 8.4 面试表达更完整

这个亮点的价值很高，因为它能体现你不是只会堆页面，而是能从：

- 系统抽象
- 可扩展性
- 配置驱动
- 注册表模式

这些角度去设计前端架构。

---

## 9. 后续档案与演进方向

后续我会继续沿着这条线做几件事。

### 9.1 抽共享题型协议

当前管理端和填写端虽然都 registry 化了，但还没有共享同一份题型协议。  
后续可以把：

- `type`
- `defaultProps`
- schema 描述

抽成共享模块。

### 9.2 扩展更多字段类型

`SchemaForm` 目前支持的字段类型还有限。  
后续可以继续加：

- number
- color
- code editor
- validation rule builder
- conditional visibility

### 9.3 后端也接入 schema 校验

现在 schema 主要用于前端属性面板。  
如果要进一步增强一致性，可以让后端在保存问卷时也基于 schema 做运行时校验。

### 9.4 题型插件化

理想状态下，新题型不应该只是“改源码新增一个目录”，而应该支持通过插件或配置注册接入。

这也是这次重构后最自然的演进方向。

## 10. 面试时怎么讲

如果让我在面试里用 1 分钟概括这个改造，我会这样说：

“原项目的题型系统只有管理端半套注册表，属性面板、统计适配和填写端渲染逻辑还是分散的，导致新增题型时要改很多地方。我把它重构成了 Schema 驱动的组件注册体系：管理端每个题型统一声明 type、分组、默认配置、属性 schema、编辑器渲染和统计渲染；右侧属性面板通过通用 SchemaForm 动态生成，不再为每个题型重复写表单组件；填写端也从 if/else 分发改成 registry 分发。这样题型能力从散装实现变成了配置驱动系统，新增题型的接入路径更清晰，也更适合后续往低代码方向演进。”

## 11. 面试官可能追问的问题

### 11.1 这算不算真正的 Schema 驱动？

算，但目前是第一阶段。  
现在已经实现了：

- 属性面板由 schema 生成
- 管理端和填写端都走 registry 分发

但还没有做到：

- 前后端完全共享 schema
- 后端基于 schema 校验

所以更准确地说，这是“向完整 Schema 驱动演进的第一步”。

### 11.2 为什么不把组件实现也抽成完全共享？

因为管理端和填写端职责不同：

- 管理端更偏预览和编辑
- 填写端更偏真实表单交互和提交

两边渲染层完全共享会带来更高耦合。  
所以这次优先统一的是：

- 元数据
- 分发方式
- 属性配置模型

### 11.3 最大的 trade-off 是什么？

最大的 trade-off 是：

- 用更抽象的 registry 和 schema，换更低的扩展成本

也就是说：

- 短期理解成本会上升一点
- 但中长期对多题型、多页面、多统计适配场景更友好

### 11.4 这个重构最核心的价值是什么？

不是少写了几个组件，而是把题型能力从“分散逻辑”升级成“系统级抽象”。  
对低代码系统来说，这一步非常关键。

---

## 12. 一句话总结

这次改造的本质，是把问卷题型从“页面里分散维护的一堆组件”升级成“由 registry 和 schema 驱动的题型系统”，让新增题型从多点散改，变成围绕统一协议的集中接入。
