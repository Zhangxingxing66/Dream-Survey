# 编辑器自动保存重构方案

## 1. 需求背景

这个项目里，问卷编辑器原先已经有“自动保存”能力，但实现方式比较简单：

- 页面内容变化后，`debounce 1s`
- 然后直接调用保存接口

这类实现对 demo 来说没问题，但如果从工程化视角看，问题比较明显：

1. 没有明确的保存状态  
   用户不知道当前内容是未保存、正在保存、保存成功，还是保存失败。

2. 高并发编辑场景下不可靠  
   如果用户连续操作，多个保存请求可能并发返回，旧请求可能比新请求更晚返回，进而覆盖更新状态。

3. 刷新或异常关闭页面会丢数据  
   如果数据还没来得及落到服务端，本地就没有任何兜底。

4. 发布前缺少一致性保障  
   用户点“发布”时，如果当前草稿还没真正保存完成，就可能发布出去旧数据。

这些问题在大厂面试里是很典型的编辑器可靠性题目。  
面试官真正关心的不是“你有没有自动保存”，而是：

- 自动保存在连续编辑下是否可靠
- 失败了怎么处理
- 刷新了能不能恢复
- 发布时怎么保证一致性

---

## 2. 需求目的

这次重构的目标不是给按钮换个文案，而是把自动保存从“能用”升级成“可靠”。

核心目标有 5 个：

1. 给保存链路引入明确状态  
   支持 `dirty / saving / saved / error`

2. 保存请求串行化  
   避免多个请求并发落库导致状态错乱

3. 丢弃过期响应  
   解决旧请求晚返回覆盖新状态的问题

4. 支持本地草稿恢复  
   避免页面刷新、关闭、接口异常导致内容丢失

5. 让发布动作依赖当前草稿真正保存成功  
   不能只看前端页面状态，必须以保存结果为准

---

## 3. 技术调研与方案对比

### 方案一：继续使用 debounce + 直接请求

思路：

- 用户修改内容
- 1 秒后触发接口
- 不做额外状态管理

优点：

- 代码最简单

缺点：

- 没有可靠性保障
- 无法处理 out-of-order response
- 刷新丢稿
- 保存失败用户无感知

结论：

- 不适合真实编辑器场景

### 方案二：自动保存 + 本地缓存，但不做请求串行化

思路：

- 保留 debounce
- 本地加 `localStorage`

优点：

- 能解决一部分刷新丢稿问题

缺点：

- 仍然无法解决多个保存请求乱序返回的问题
- UI 状态和真实服务端状态可能不一致

结论：

- 只解决了一半问题

### 方案三：状态机 + 串行保存队列 + 过期响应丢弃 + 本地草稿恢复

思路：

- 自动保存不再只是“调接口”
- 而是被当成一个独立的保存系统

包含：

- 明确状态
- 请求串行化
- 基线快照
- 本地恢复
- 发布前强制保存

优点：

- 可靠性更强
- 容易解释和扩展
- 面试可讲深度高

缺点：

- 实现复杂度明显增加

最终选择：

- 采用方案三

因为编辑器类应用的核心价值之一就是“数据不能乱”，这个问题不能靠简单 debounce 硬扛。

---

## 4. 最终设计方案

## 4.1 核心思想

把自动保存从“事件触发请求”升级成“受控的保存状态机”。

一句话概括：

`当前编辑态 - 基线快照 - 串行保存队列 - 本地恢复`

### 4.2 状态设计

我把自动保存状态抽象成：

- `idle`
- `dirty`
- `saving`
- `saved`
- `error`

其中面试重点其实是后四个：

#### dirty

表示当前本地编辑态和上次成功保存的快照不一致。

#### saving

表示当前有保存任务正在处理。

#### saved

表示当前本地状态已经和服务端基线一致。

#### error

表示最近一次保存失败，并且当前快照仍未成功落库。

这样设计的意义是：

- UI 状态能真实反映保存系统，而不只是某次按钮点击

---

## 5. 实现细节

## 5.1 保存链路抽成独立 hook

我把逻辑抽到了：

- `wenjuan-fe/src/hooks/useQuestionAutoSave.ts`

这样做有两个好处：

1. 编辑器头部不再堆积保存细节
2. 自动保存逻辑可以被单独理解和迭代

它的输入主要是：

- 当前问卷 id
- 页面加载状态
- 当前 pageInfo
- 当前 componentList

输出包括：

- `status`
- `errorMessage`
- `isDirty`
- `saveNow`

### 5.2 基线快照设计

这是整个方案里最重要的概念之一。

我会把当前编辑器数据抽成统一 payload：

```ts
{
  title,
  desc,
  js,
  css,
  componentList
}
```

然后序列化成 `snapshot`。

系统内部会维护两份关键快照：

1. `latestSnapshot`  
   当前编辑器最新内容

2. `lastSavedSnapshot`  
   最近一次成功保存到服务端的内容

判断是否脏数据，本质上就是：

- `latestSnapshot !== lastSavedSnapshot`

这样做的好处是：

- 不依赖某个局部字段
- 可以统一覆盖页面设置和组件树变化

### 5.3 请求串行队列

这次重构里，我没有让每次编辑都直接发请求，而是引入了：

- `queuedPayloadRef`
- `processingPromiseRef`

逻辑上等价于一个单消费者保存队列：

1. 有新草稿时，先进入队列
2. 如果当前没有请求在处理，就开始保存
3. 如果当前已有请求在处理，只更新队列中的“待保存最新值”
4. 当前请求结束后，再处理队列中的下一份最新草稿

关键点在于：

- 队列里只保留“最新待保存草稿”
- 而不是把每一次编辑都完整排队

因为对问卷编辑器来说，中间过渡状态没必要全部落库，真正重要的是：

- 最终最新状态能可靠保存

### 5.4 过期响应丢弃机制

这是最容易被面试官追问的一点。

问题场景：

1. 第 1 次请求先发出
2. 第 2 次请求后发出
3. 但第 1 次请求更晚返回

如果没有保护，就会出现：

- 旧响应覆盖新状态

我这里做了两层保护：

#### 第一层：保存串行化

先从源头上减少并发请求。

#### 第二层：响应有效性校验

每次保存请求都会带自己的上下文：

- `requestId`
- `sessionId`
- `questionId`

响应回来时会检查：

1. 是否还是当前编辑会话
2. 是否还是当前问卷
3. `requestId` 是否落后于已经应用过的请求

只要任一条件不满足，就直接判定为 stale response，丢弃响应结果。

这一步的意义是：

- 即便未来某些场景下出现并发或切页，也不会让旧响应污染当前状态

### 5.5 本地草稿恢复

我新增了：

- `wenjuan-fe/src/utils/question-draft.ts`

职责包括：

- 生成本地草稿 payload
- 序列化快照
- 写入 `localStorage`
- 读取和删除本地草稿

恢复策略是这样的：

1. 问卷服务端草稿加载完成后，检查本地是否存在该问卷未保存草稿
2. 如果本地快照和服务端当前快照一致，直接删除本地缓存
3. 如果不一致，弹窗询问用户是否恢复本地草稿

这里我没有直接自动覆盖服务端数据，而是让用户做一次显式选择。  
原因是：

- 本地草稿不一定比服务端更新更可信
- 自动覆盖风险太高

### 5.6 服务端基线同步

自动保存系统必须知道“当前服务端基线什么时候变了”。

所以我把：

- `updatedAt`

同步进了前端页面状态：

- `pageInfoReducer`
- `question-store`

只要检测到服务端 `updatedAt` 更新，就说明：

- 当前这份草稿已经有新的服务端基线

这时候会：

1. 更新 `baselineServerUpdatedAt`
2. 更新 `lastSavedSnapshot`
3. 清掉本地临时草稿

这样回滚、发布或重新加载草稿后，自动保存系统都能正确重置基线。

### 5.7 发布前强制保存

原先发布逻辑是：

- 先调一次更新接口
- 再调发布接口

现在改成：

- 调用 `saveNow()`
- 只有当前草稿真正保存成功，才允许执行发布

如果保存失败：

- 发布直接取消

这一步是保证“发布的一定是当前页面看到的内容”的关键。

## 6. 遇到的坑

### 6.1 初始化阶段误触发自动保存

这是第一个坑。

编辑页加载时，服务端草稿会写入 Redux。  
如果自动保存逻辑不区分“首次加载”和“用户编辑”，就会出现：

- 页面刚打开就被判定为 dirty
- 然后触发一次多余保存

解决方法：

- 引入 `initializedQuestionIdRef`
- 只有当前问卷完成初始化后，才开始自动保存逻辑

### 6.2 本地草稿恢复和服务端草稿冲突

第二个坑是：

- 本地草稿和服务端草稿不一致时，到底自动恢复还是让用户选？

最终选择：

- 弹窗确认

原因：

- 自动覆盖过于激进
- 用户需要明确知道自己是在恢复本地未保存内容

### 6.3 保存失败后是否继续自动重试

第三个坑是：

- 如果某次保存失败，要不要继续自动定时重试？

这次我没有做“无限自动重试”。

因为无限重试会有两个风险：

1. 接口异常时产生无意义请求风暴
2. 用户看不到清晰错误边界

当前策略是：

- 当前快照失败后进入 `error`
- 用户继续编辑形成新快照后，再重新进入自动保存流程
- 或者用户手动点保存重试

这更符合可控性原则。

### 6.4 手动保存和自动保存如何协同

如果用户点击“保存草稿”，但当前队列里已经有自动保存任务，逻辑很容易互相打架。

我最终统一成：

- 不管是自动保存、快捷键保存还是按钮保存，最终都走同一个 `saveNow()`

这样避免了两套保存逻辑分叉。

## 7. 底层原理

这次重构背后的核心原理有 4 个。

### 7.1 基线快照比较

不是依赖某个 `dirty = true/false` 标记手动维护，而是通过：

- 当前快照
- 最近成功保存快照

进行对比判断。

这是一个更稳健的方式，因为它不容易被局部状态遗漏。

### 7.2 单消费者保存队列

保存系统不是“每次编辑都立刻请求”，而是：

- 最新草稿入队
- 队列按顺序消费

它本质上是一个简化版的单消费者任务队列。

### 7.3 会话隔离和过期响应防御

通过：

- `sessionId`
- `requestId`
- `questionId`

三重约束，保证旧请求不会影响新会话或新状态。

这类设计在编辑器、协作系统、流式请求里都很常见。

### 7.4 本地持久化兜底

服务端保存负责最终一致性。  
本地 `localStorage` 负责编辑过程中的灾难恢复。

两者职责分离：

- 服务端是正式存储
- 本地缓存是编辑兜底

---

## 8. 收益

这次重构带来的收益可以从 4 个方面讲。

### 8.1 编辑体验更可感知

用户能直接看到：

- 当前有没有未保存改动
- 是否正在保存
- 是否保存成功
- 是否发生错误

### 8.2 高并发编辑场景更可靠

通过保存串行化和过期响应丢弃，解决了连续编辑下的数据错乱风险。

### 8.3 数据丢失风险下降

本地草稿恢复机制让刷新、异常关闭、不稳定网络下的丢稿问题有了兜底。

### 8.4 发布一致性更强

发布动作依赖真实保存成功，不会出现“页面看到的是新内容，线上发布的是旧内容”的情况。

---

## 9. 后续档案与演进方向

后续我会继续沿着这个保存系统做几类扩展。

### 9.1 页面离开保护

如果当前 `isDirty = true`，在用户关闭页面或跳转时提示：

- 仍有未保存内容

### 9.2 自动重试策略

对网络抖动类错误，可以引入：

- 指数退避重试

但要限制次数，避免请求风暴。

### 9.3 服务端乐观锁

如果未来支持多人编辑，需要在服务端引入：

- 版本号
- 乐观锁

否则只有前端队列还不够。

### 9.4 更细粒度的本地恢复

当前是按整份问卷 payload 恢复。  
后续可以扩展为：

- 自动记录恢复时间
- 提供对比恢复
- 支持多份本地历史草稿

---

## 10. 面试时怎么讲

如果让我用 1 分钟概括这个优化，我会这样说：

“原来的自动保存只是 debounce 后直接调保存接口，在连续编辑场景下缺少可靠性保障。我把它重构成了独立的自动保存系统：一方面引入 `dirty / saving / saved / error` 状态，让 UI 能反映真实保存过程；另一方面通过单消费者保存队列和基于 `requestId / sessionId / questionId` 的过期响应丢弃机制，解决旧请求晚返回导致状态错乱的问题；同时把未保存草稿落到 `localStorage`，在重新进入编辑页时支持用户恢复本地草稿。最后发布动作也改成依赖 `saveNow()` 成功后才允许继续，确保发布内容和当前编辑内容一致。”

## 11. 面试官可能追问的问题

### 11.1 为什么不用 WebSocket 实时同步？

这个场景本质是单人编辑器自动保存，不是多人协同。  
WebSocket 会明显增加系统复杂度，但收益不大。  
当前问题的关键不是“实时”，而是“可靠保存”。

### 11.2 为什么用快照对比，而不是简单 dirty flag？

因为简单 dirty flag 容易在复杂状态更新里失真。  
快照对比更稳健，而且可以自然兼容页面设置和组件树的整体变化。

### 11.3 为什么不把每次修改都完整排队保存？

因为中间态没有业务价值。  
问卷编辑器关心的是最终最新状态能可靠保存，而不是每一步中间编辑动作都持久化。

### 11.4 这套方案能直接支持多人协同吗？

不能直接支持。  
它解决的是单客户端下的保存一致性问题。  
多人协同还需要：

- 服务端版本控制
- 冲突检测
- 乐观锁或 OT/CRDT

### 11.5 最大的 trade-off 是什么？

最大的 trade-off 是：

- 用更复杂的状态和引用管理，换取更可靠的数据保存能力

也就是说：

- 代码复杂度提升了
- 但这对编辑器类系统是值得的

---

## 12. 一句话总结

这次改造的本质，是把问卷编辑器的自动保存从“一个 debounce 接口调用”升级成了“有状态、有队列、有过期响应防御、有本地恢复兜底的可靠保存系统”。
