# 统计模块增量聚合方案

## 1. 需求背景

原始项目的统计模块有两个接口：

- `GET /api/stat/:questionId`
- `GET /api/stat/:questionId/:componentId`

其中：

- 分页明细接口会读取当前版本下的所有答卷，再做分页和字段映射
- 图表统计接口会扫描当前版本下的所有答卷，然后针对单选、多选题做全量聚合

这种实现对数据量小时没问题，但从工程化角度看，有明显瓶颈：

1. 图表统计是典型的重复计算  
   每次打开统计页、每次切换题目，都要重新扫描一遍答卷。

2. 统计耗时会随答卷量线性增长  
   问卷答卷越多，图表接口越慢。

3. 题型统计逻辑写死在接口里  
   目前只有单选、多选支持图表统计，后续新增题型时扩展路径不清晰。

4. 已经做了版本化发布之后，统计更应该绑定到发布版本  
   如果继续请求时现算，读时开销会持续扩大。

所以这个需求的本质不是“把接口写快一点”，而是把统计从“读时全量计算”升级成“写时增量维护 + 组件级适配”。

---

## 2. 需求目的

这次改造的目标有 4 个：

1. 把图表统计从请求时现算改成提交答卷时增量维护  
   降低图表统计接口的重复计算成本。

2. 抽象组件级 `statAdapter`  
   让不同题型的统计规则通过统一协议接入，而不是散落在接口逻辑里。

3. 和版本化发布模型对齐  
   每个发布版本拥有自己的统计快照，避免跨版本混算。

4. 保持明细能力不丢  
   分页答卷明细仍然保留请求时读取，因为它本质是列表查询，不适合完全预聚合。

---

## 3. 技术调研与方案对比

### 方案一：继续请求时全量扫描

思路：

- 每次请求统计接口时，从 `answers` 里扫描当前版本所有答卷
- 现算结果后返回

优点：

- 实现简单
- 不需要维护额外聚合结构

缺点：

- 图表接口重复扫描答卷
- 数据量增长后性能持续退化
- 不利于题型扩展

结论：

- 适合 demo，不适合真实业务演进

### 方案二：所有统计全部预聚合

思路：

- 明细列表和图表统计都在写入答卷时预处理

优点：

- 读接口会非常快

缺点：

- 明细列表本质是原始答卷浏览，不适合做完全聚合
- 写路径复杂度会明显上升
- 聚合结构会更重

结论：

- 过度设计，不适合当前项目阶段

### 方案三：图表统计增量聚合，明细列表保留读时分页

思路：

- 图表统计：提交答卷时维护版本级聚合结果
- 明细列表：保留请求时读取原始答卷

优点：

- 解决最核心的性能问题
- 改造范围可控
- 模型更符合统计场景

缺点：

- 写路径变复杂
- 需要处理旧数据兼容

最终选择：

- 采用方案三

原因：

- 图表统计是最适合做写时聚合的部分
- 明细列表是典型读原始数据场景，没必要强行预计算

---

## 4. 最终设计方案

## 4.1 核心思想

把统计拆成两类：

1. 原始明细统计  
   保留答卷列表，按请求分页读取

2. 聚合图表统计  
   在答卷提交时同步维护

一句话概括：

`明细保留原始答卷，图表改为写时聚合`

### 4.2 聚合单位为什么选“发布版本”

当前项目已经做了“草稿态 / 发布态”分离。  
所以统计天然应该挂在：

- 问卷 id
- 发布版本 id

而不是只挂在问卷 id 上。

原因很直接：

- 同一问卷不同发布版本，题目结构和选项可能不同
- 如果把所有答卷混在一起统计，会破坏统计语义

因此这次的聚合粒度是：

- `questionId + versionId`

### 4.3 数据模型设计

我把聚合结果放在发布版本快照上：

```ts
{
  id,
  versionNumber,
  title,
  desc,
  componentList,
  answerCount,
  statSummary
}
```

其中：

- `answerCount`：该版本下的答卷数量
- `statSummary`：该版本下各可统计题型的聚合结果

`statSummary` 结构大致如下：

```ts
{
  [componentId]: {
    componentId,
    type,
    entries: [
      { value, name, count }
    ]
  }
}
```

这意味着：

- 每个发布版本拥有一份独立的聚合统计快照

## 5. 实现细节

## 5.1 组件级 statAdapter 设计

核心实现放在：

- `wenjuan-server/index.js`

我引入了一套服务端 `statAdapterMap`，目前支持：

- `questionRadio`
- `questionCheckbox`

每个 adapter 统一实现三件事：

1. `createEntries(component)`  
   根据组件配置生成初始统计桶

2. `accumulate(summary, rawValue)`  
   在提交答卷时，把一条答案增量累加到聚合结果

3. `toStat(summary)`  
   把内部聚合结构转换成前端图表需要的格式

这样设计的好处是：

- 统计规则不再散落在接口里
- 新题型接入图表统计时，只需要补一个 adapter

### 5.2 发布时初始化统计结构

每次发布问卷时，都会生成一个新版本快照。  
这次改造后，发布快照除了原有内容外，还会额外初始化：

- `answerCount: 0`
- `statSummary`

初始化函数是：

- `createVersionStatSummary(componentList)`

它会遍历组件列表，只为有统计能力的组件建立聚合桶。

### 5.3 写时增量维护

答卷提交接口：

- `POST /api/answer`

现在除了写入原始答卷，还会做两件事：

1. `version.answerCount += 1`
2. `applyAnswerToVersionStats(version, answerRecord)`

也就是说：

- 原始答案仍然保留
- 聚合结果同步更新

这就是典型的写时聚合。

### 5.4 图表统计接口改造

原先：

- `/api/stat/:questionId/:componentId`

会扫描当前版本下所有答卷，再现算单组件统计。

现在：

- 直接读取版本快照上的 `statSummary`
- 再通过对应 `statAdapter.toStat()` 转成前端图表数据

这意味着图表接口的复杂度从：

- 依赖答卷量的线性扫描

变成：

- 直接读取版本聚合结果

### 5.5 明细统计保留原始答卷读取

`/api/stat/:questionId` 这类接口本质上是答卷明细列表。  
它不是简单 count，而是：

- 分页
- 展示每条答卷的字段值

所以这部分我没有强行做预聚合，而是：

- 仍然读取原始答卷列表
- 但总数优先使用版本级 `answerCount`

这样设计的好处是：

- 性能优化点集中在图表统计
- 明细接口仍然保持简单可理解

### 5.6 旧数据兼容与迁移

这个改造不能只考虑新数据，因为项目里已经有旧版本数据。

所以在 `normalizeDb()` 里，我补了兼容逻辑：

1. 如果旧版本没有 `answerCount` 或 `statSummary`
2. 则读取该版本对应的历史答卷
3. 回放一次 `applyAnswerToVersionStats`
4. 重建出完整聚合结果

这一步的价值非常大，因为它保证了：

- 老问卷不需要手动迁移脚本
- 服务启动后就能平滑补齐统计结构

## 6. 遇到的坑

### 6.1 统计应该挂在问卷上还是版本上

这是第一个关键决策点。

如果统计只挂在问卷上：

- 版本化发布后，不同版本的答卷会混在一起

如果统计挂在版本上：

- 每次发布都是一份新的统计上下文

最终我选择挂在版本上。  
因为版本才是稳定的业务快照。

### 6.2 是否要把所有统计都做成增量聚合

第二个坑是：

- 明细列表要不要也一起预聚合？

答案是不需要。

因为明细列表本质是原始数据浏览，不是聚合视图。  
如果为了追求“全预聚合”而把它也做复杂，收益不高，反而增加维护成本。

### 6.3 旧数据兼容

第三个坑是：

- 老版本快照里原本没有 `statSummary`

如果不兼容处理，发布后的旧问卷会直接失去图表统计能力。

所以必须在读库时做一次“增量聚合回放式迁移”。

### 6.4 提交路径变重了

写时聚合的代价是：

- 每次答卷提交，不再只是 append 一条记录
- 还要更新版本聚合结构

这是一个典型的 trade-off：

- 用写入时的少量额外开销，换读取时的大幅稳定性

对于问卷这种“读统计很多、写答卷相对少”的场景，这个 trade-off 是合理的。

## 7. 底层原理

### 7.1 写时聚合

这次的核心原理是：

- 把部分统计计算从“读接口时做”
- 前移到“写答卷时做”

这是典型的：

- 以写换读

在统计分析、报表、埋点系统里都非常常见。

### 7.2 组件级适配器模式

不同题型的统计逻辑不同：

- 单选是“一票投给一个选项”
- 多选是“一份答卷可能投给多个选项”

如果继续把逻辑都写死在接口里，后续会越来越混乱。  
所以这次把统计规则抽象成题型 adapter：

- 创建桶
- 累加值
- 转出格式

这就是典型的适配器模式。

### 7.3 版本隔离

统计挂在发布版本上，本质上是在做：

- 数据上下文隔离

这能保证：

- 每个版本的图表统计只解释该版本看到的题目结构

这是版本化问卷系统里很关键的一点。

---

## 8. 收益

这次改造带来的收益主要有 4 个。

### 8.1 图表统计读接口性能更稳定

之前每次切换组件都要重新扫描答卷。  
现在图表统计直接读聚合结果，接口性能更稳定。

### 8.2 新题型扩展路径更清晰

如果后续要让某个题型支持图表统计，只需要补一个 `statAdapter`，而不是改接口主体逻辑。

### 8.3 和版本发布体系形成闭环

现在：

- 题型 schema 负责定义组件能力
- 版本发布负责冻结线上结构
- 增量聚合负责维护统计快照

这三者在模型层面是打通的。

### 8.4 面试价值高

这个优化很适合大厂面试，因为它能体现：

- 你不只是会写 CRUD
- 还会考虑性能模型、读写 trade-off、适配器抽象和兼容迁移

## 9. 后续档案与演进方向

### 9.1 按版本切换查看统计

当前底层已经支持版本级统计。  
后续可以在统计页增加版本切换 UI，支持查看历史发布版本的聚合结果。

### 9.2 更多题型接入 statAdapter

现在只有：

- 单选
- 多选

后续可以继续支持：

- 评分题
- 下拉选择
- 矩阵题

### 9.3 聚合和明细进一步解耦

如果数据规模继续上升，可以把：

- 原始答卷
- 聚合统计

进一步拆成独立存储结构，而不是都放在同一 JSON 模型中。

### 9.4 导出与趋势分析

当前只做了静态聚合。  
后续还可以扩展：

- 时间维度趋势
- 导出 CSV
- 多维交叉统计

---

## 10. 面试时怎么讲

如果让我用 1 分钟概括这次优化，我会这样说：

“原项目的统计页图表接口每次都会全量扫描当前版本下的答卷来现算单选、多选统计，数据量一大就会线性退化。我把它改成了组件级 statAdapter + 增量聚合方案：发布问卷时会在版本快照上初始化 `statSummary` 统计桶，提交答卷时除了写原始答案，还会同步维护该发布版本的 `answerCount` 和 `statSummary`；图表统计接口不再扫描全量答卷，而是直接读取版本聚合结果，再由 adapter 转成前端图表数据。这样把图表统计从读时计算改成了写时聚合，提升了性能稳定性，也让后续题型扩展有了统一接入方式。”

## 11. 面试官可能追问的问题

### 11.1 为什么不把明细列表也做成预聚合？

因为明细列表是原始数据浏览，不是聚合视图。  
它更适合保留原始答卷并做分页查询。  
这次优化的核心对象是图表统计，而不是所有统计都一刀切预计算。

### 11.2 为什么不直接上数据库做聚合查询？

当前项目后端是本地 JSON 文件存储。  
这次重点不是数据库选型，而是先把统计模型建对。  
如果迁移到真实数据库，`statAdapter` 和版本级聚合思路仍然成立，只是落地形态会从 JSON 变成表结构或缓存结构。

### 11.3 写时聚合会不会影响提交性能？

会增加一点写入成本，但这是有意的 trade-off。  
问卷场景通常是：

- 图表查询频次高
- 答卷提交相对低频

所以用少量写开销换更稳定的读性能是合理的。

### 11.4 如果后续题目选项变了怎么办？

因为统计是挂在发布版本上的，版本快照里已经固化了当时的组件结构和选项。  
所以后续草稿改动不会污染当前版本统计。

### 11.5 最大的收益是什么？

最大的收益不是把某个接口提速，而是把统计模块从“接口里临时现算”升级成了“和题型、版本一起演进的可扩展聚合架构”。

---

## 12. 一句话总结

这次改造的本质，是把问卷图表统计从“读时全量扫描答卷”升级成“按发布版本维护的组件级增量聚合系统”，核心价值在于性能稳定性和统计能力的可扩展性。
