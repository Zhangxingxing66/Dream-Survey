# Dream Survey 项目学习指南

## 1. 先说结论

这个仓库不是单一应用，而是 3 个相互配合的子项目：

- `wenjuan-fe`：问卷管理端，负责登录、问卷列表、编辑器、发布、统计
- `wenjuan-client`：问卷填写端，负责展示已发布问卷并提交答案
- `wenjuan-server`：本地后端，基于 Koa + JSON 文件持久化

理解这个项目，最有效的方式不是按目录平铺去看，而是按一条完整业务链路去追：

1. 管理端登录
2. 创建问卷
3. 编辑问卷组件并自动保存
4. 发布问卷
5. 填写端根据问卷配置渲染页面并提交答案
6. 管理端读取统计结果

如果你只想快速抓住主干，优先看这几个文件：

- `wenjuan-server/index.js`
- `wenjuan-fe/src/router/index.tsx`
- `wenjuan-fe/src/pages/question/Edit/index.tsx`
- `wenjuan-fe/src/store/componentsReducer/index.ts`
- `wenjuan-fe/src/components/QuestionComponents/index.ts`
- `wenjuan-client/src/pages/question/[id].tsx`
- `wenjuan-client/src/components/QuestionComponents/index.tsx`

## 2. 项目整体架构

### 2.1 角色分工

`wenjuan-fe`

- 面向问卷创建者
- 负责用户登录、问卷管理、拖拽编辑、发布、统计分析
- 技术栈：React 18 + React Router 6 + Redux Toolkit + Ant Design + ahooks + dnd-kit

`wenjuan-client`

- 面向问卷填写者
- 根据问卷 JSON 动态渲染表单
- 提交表单后跳转成功/失败页
- 技术栈：Next.js 13 Pages Router + SSR + Sass

`wenjuan-server`

- 提供所有真实 API
- 不依赖数据库，直接把数据写到 `data/db.json`
- 技术栈：Koa + koa-router + Node `fs/promises`

### 2.2 请求关系

```text
管理端 wenjuan-fe:8000
  └─ /api/* 通过 craco devServer proxy 转发到 3001

填写端 wenjuan-client:3000
  └─ 服务端渲染阶段/Next API Route 直接请求 http://localhost:3001

后端 wenjuan-server:3001
  └─ 读写 data/db.json
```

### 2.3 一句话理解设计思路

核心思想是“问卷 = 页面信息 + 组件列表”。  
编辑器维护组件列表，后端原样保存；填写端再按组件类型把这份配置渲染回来。

## 3. 怎么跑起来

建议开 3 个终端，按下面顺序启动：

### 3.1 启动后端

```bash
cd wenjuan-server
npm install
npm run dev
```

默认端口：`3001`

### 3.2 启动管理端

```bash
cd wenjuan-fe
npm install
npm start
```

默认端口：`8000`

说明：

- `craco.config.js` 里把 `/api` 代理到了 `http://localhost:3001`

### 3.3 启动填写端

```bash
cd wenjuan-client
npm install
npm run dev
```

默认端口：`3000`

说明：

- `src/services/ajax.ts` 默认请求 `http://localhost:3001`
- 首页 `src/pages/index.tsx` 里默认把管理端地址写成了 `http://localhost:8000`

## 4. 先看后端，再看前端

这是这个项目最高效的阅读顺序。

原因很简单：

- 后端定义了真实数据结构
- 管理端和填写端本质上都在消费这份结构
- 只要先弄清 `question` 长什么样，后面很多代码会瞬间变简单

## 5. 后端学习主线

核心文件：`wenjuan-server/index.js`

### 5.1 后端做了什么

它把所有逻辑都放在一个文件里，包含：

- 启动 Koa 服务
- CORS 处理
- 请求体解析
- JSON 文件读写
- 登录态校验
- 问卷 CRUD
- 答卷提交
- 统计计算

### 5.2 数据落在哪里

真实数据都在：

- `wenjuan-server/data/db.json`

里面有 4 张“表”：

- `users`
- `sessions`
- `questions`
- `answers`

### 5.3 最重要的数据模型

#### user

```ts
{
  id,
  username,
  password,
  nickname,
  createdAt
}
```

#### session

```ts
{
  token,
  userId,
  createdAt
}
```

#### question

```ts
{
  id,
  userId,
  title,
  desc,
  js,
  css,
  isDeleted,
  isPublished,
  isStar,
  answerCount,
  createdAt,
  updatedAt,
  componentList
}
```

#### answer

```ts
{
  id,
  questionId,
  createdAt,
  answerList: [
    {
      componentId,
      value
    }
  ]
}
```

### 5.4 问卷的本质

问卷不是后端硬编码页面，而是一段配置：

- 页面级信息：`title`、`desc`、`js`、`css`
- 内容级信息：`componentList`

`componentList` 里的每一项都像这样：

```ts
{
  fe_id,
  type,
  title,
  isHidden,
  isLocked,
  props
}
```

也就是说，前端编辑器其实是在编辑一棵组件配置树。

### 5.5 先读这几组函数

先读“底层通用能力”：

- `readDb`
- `writeDb`
- `updateDb`
- `getCurrentUser`
- `requireUser`

再读“问卷模型生成”：

- `createDefaultComponentList`
- `createQuestion`
- `resetComponentIds`

最后读“统计逻辑”：

- `buildStatRow`
- `buildComponentStat`

### 5.6 重要接口

#### 用户

- `POST /api/user/register`
- `POST /api/user/login`
- `GET /api/user/info`

#### 问卷

- `POST /api/question`：创建问卷
- `GET /api/question/:id`：获取单个问卷
- `GET /api/question`：分页查列表
- `PATCH /api/question/:id`：更新问卷
- `POST /api/question/duplicate/:id`：复制问卷
- `DELETE /api/question`：批量彻底删除

#### 答卷和统计

- `POST /api/answer`
- `GET /api/stat/:questionId`
- `GET /api/stat/:questionId/:componentId`

### 5.7 后端值得注意的实现点

- 没有数据库，`updateDb` 用 `writeChain` 串行化写入，避免并发写文件冲突
- 登录态不是 JWT，而是简单 Bearer Token + `sessions` 数组
- 统计接口不是实时 SQL 聚合，而是每次从 `answers` 重新计算
- `normaliseQuestion` 和 `normaliseQuestionListItem` 在兼容前端字段格式，比如 `_id`

## 6. 管理端学习主线

管理端目录：`wenjuan-fe`

### 6.1 先看入口

按这个顺序看：

1. `src/index.tsx`
2. `src/App.tsx`
3. `src/router/index.tsx`
4. `src/layouts/MainLayout.tsx`
5. `src/layouts/QuestionLayout.tsx`

你会先看清：

- React 应用如何挂载
- Redux Provider 怎么注入
- 路由如何拆成普通页面和问卷工作台页面
- 用户信息什么时候加载
- 未登录时如何跳转

### 6.2 路由结构

主路由大致如下：

```text
/
├─ /                首页
├─ /login           登录
├─ /register        注册
├─ /manage/list     我的问卷
├─ /manage/star     星标问卷
├─ /manage/trash    回收站
├─ /question/edit/:id  编辑器
└─ /question/stat/:id  统计页
```

其中：

- `Edit`
- `Stat`

用了 `React.lazy` 懒加载，说明作者考虑过 bundle 拆分。

### 6.3 登录态流转

相关文件：

- `src/hooks/useLoadUserData.ts`
- `src/hooks/useNavPage.ts`
- `src/services/user.ts`
- `src/services/ajax.ts`
- `src/utils/user-token.ts`
- `src/store/userReducer.ts`

流程：

1. 页面进入 `MainLayout` 或 `QuestionLayout`
2. `useLoadUserData` 请求 `/api/user/info`
3. 用户信息写入 Redux
4. `useNavPage` 根据当前路径和登录态决定是否跳转
5. 所有 axios 请求自动带 `Authorization: Bearer <token>`

### 6.4 列表页怎么读

先看：

- `src/pages/manage/List.tsx`
- `src/hooks/useLoadQuestionListData.ts`
- `src/services/question.ts`

你会发现两个不同风格：

- `useLoadQuestionListData.ts` 是标准分页查询 hook
- `List.tsx` 自己又实现了一套“滚动到底加载更多”

这说明项目在列表页上存在一些“演进痕迹”，学习时不要纠结成必须绝对统一的架构。

### 6.5 编辑器是项目核心

先按这个顺序看：

1. `src/pages/question/Edit/index.tsx`
2. `src/hooks/useLoadQuestionData.ts`
3. `src/store/componentsReducer/index.ts`
4. `src/store/pageInfoReducer.ts`
5. `src/hooks/useGetComponentInfo.ts`
6. `src/hooks/useGetPageInfo.ts`
7. `src/pages/question/Edit/EditHeader.tsx`
8. `src/pages/question/Edit/EditCanvas.tsx`
9. `src/pages/question/Edit/LeftPanel.tsx`
10. `src/pages/question/Edit/RightPanel.tsx`

### 6.6 编辑器的状态分层

这个项目把编辑器状态分成两部分：

#### 组件级状态

放在 `componentsReducer`

- `componentList`
- `selectedId`
- `copiedComponent`

负责：

- 新增组件
- 删除组件
- 修改组件 props
- 隐藏/显示
- 锁定/解锁
- 复制/粘贴
- 上下选择
- 拖拽排序

#### 页面级状态

放在 `pageInfoReducer`

- `title`
- `desc`
- `js`
- `css`
- `isPublished`

负责问卷整体属性，而不是单个题目组件。

### 6.7 为什么有 `redux-undo`

`src/store/index.ts` 里对 `componentsReducer` 包了一层 `undoable(...)`。

作用：

- 给编辑器提供撤销/重做能力
- 并且排除了选中切换之类不值得进入历史栈的 action

这是编辑器类应用的典型设计点，值得重点看。

### 6.8 组件注册表是另一个核心

关键文件：

- `src/components/QuestionComponents/index.ts`

它不是简单导出组件，而是在维护“组件配置中心”：

- `title`
- `type`
- `Component`
- `PropComponent`
- `defaultProps`
- `StatComponent`

这意味着一个题型在管理端通常有 3 种视图：

1. 画布上的渲染组件 `Component`
2. 右侧属性面板组件 `PropComponent`
3. 统计页图表组件 `StatComponent`

再加上：

- `componentConfGroup`

用于左侧组件库分组展示。

### 6.9 新增组件的大致改动面

如果以后你要扩展一个新题型，通常至少要改：

管理端：

- `src/components/QuestionComponents/你的组件/Component.tsx`
- `src/components/QuestionComponents/你的组件/PropComponent.tsx`
- `src/components/QuestionComponents/你的组件/interface.ts`
- `src/components/QuestionComponents/你的组件/index.ts`
- `src/components/QuestionComponents/index.ts`

填写端：

- `wenjuan-client/src/components/QuestionComponents/你的组件`
- `wenjuan-client/src/components/QuestionComponents/index.tsx`

后端：

- 如果只是普通文本输入，通常不用改
- 如果要支持统计图，可能需要改 `buildComponentStat`

### 6.10 自动保存机制

在 `src/pages/question/Edit/EditHeader.tsx` 里：

- `useDebounceEffect` 会在 `componentList` 或 `pageInfo` 变化后延迟 1 秒自动保存
- `Ctrl + S` 也会触发保存
- 发布本质上也是一次 `PATCH /api/question/:id`，只是额外把 `isPublished` 设为 `true`

这是理解“编辑页数据何时落库”的关键点。

## 7. 填写端学习主线

填写端目录：`wenjuan-client`

### 7.1 最重要的页面

先看：

1. `src/pages/question/[id].tsx`
2. `src/components/QuestionComponents/index.tsx`
3. `src/components/PageWrapper.tsx`
4. `src/pages/api/answer.ts`
5. `src/services/question.ts`
6. `src/services/answer.ts`

### 7.2 页面渲染链路

`[id].tsx` 的逻辑非常直接：

1. `getServerSideProps` 根据问卷 id 请求后端
2. 判断是否报错、是否删除、是否已发布
3. 遍历 `componentList`
4. 根据 `type` 调 `getComponent(c)`
5. 生成表单
6. 提交到 Next 自己的 `/api/answer`

这里很值得你注意：

- 填写端不是直接把表单提交给 Koa
- 它先提交给 Next API Route
- 再由 `src/pages/api/answer.ts` 转成统一的 `answerList`
- 最终转发到后端的 `POST /api/answer`

### 7.3 为什么要经过 Next API Route

因为浏览器原生 form 提交的数据格式不是后端最终想要的：

原始表单是：

```ts
{
  questionId,
  [componentId]: value
}
```

而后端要的是：

```ts
{
  questionId,
  answerList: [
    { componentId, value }
  ]
}
```

`src/pages/api/answer.ts` 就是在做这个转换。

### 7.4 填写端和管理端的对应关系

管理端负责产出组件配置。  
填写端负责消费组件配置。

所以学习填写端时，建议始终和管理端的组件配置中心对照着看：

- 管理端的组件 `type`
- 填写端的 `if (type === 'questionInput') ...`

这样最容易建立全局感。

### 7.5 页面级自定义能力

`PageWrapper.tsx` 会把问卷的：

- `css`
- `js`

直接注入页面。

这说明问卷支持一定程度的页面级定制，但也意味着：

- 安全性几乎没有隔离
- 更像本地教学/练习项目，不是生产级沙箱设计

## 8. 一条完整业务链路怎么走

### 8.1 注册和登录

管理端：

- `Login/Register` 页面提交账号密码
- 后端写入 `users`
- 登录成功后写入 `sessions`
- 前端把 token 存本地

### 8.2 创建问卷

管理端调用：

- `POST /api/question`

后端：

- 根据 `createQuestion(userId)` 生成一份默认问卷
- 自动带默认组件列表

### 8.3 编辑问卷

编辑页进入后：

1. `useLoadQuestionData` 拉取问卷详情
2. 把 `componentList` 塞进 `componentsReducer`
3. 把 `title/desc/js/css/isPublished` 塞进 `pageInfoReducer`
4. 用户操作 Redux 状态
5. 自动保存触发 `PATCH /api/question/:id`

### 8.4 发布问卷

管理端点击发布：

- 更新 `isPublished: true`
- 跳转到统计页

### 8.5 填写问卷

填写端访问：

- `/question/:id`

服务端渲染拿到问卷配置，按组件类型动态生成页面。

### 8.6 提交答卷

填写端表单提交：

1. 到 Next API Route `/api/answer`
2. 转成 `answerList`
3. 转发给 Koa `/api/answer`
4. 后端写入 `answers`
5. 同时更新 `question.answerCount`

### 8.7 查看统计

管理端统计页分两部分：

- 表格统计：`GET /api/stat/:questionId`
- 单组件图表统计：`GET /api/stat/:questionId/:componentId`

## 11. 这个项目的几个关键设计点

### 11.1 优点

- 业务主线清晰，适合学习“低代码问卷”的核心思想
- 组件配置驱动页面渲染，抽象层次是对的
- 管理端和填写端职责分离明显
- 本地 JSON 持久化让调试成本很低
- 编辑器具备撤销/重做、自动保存、拖拽排序等典型能力

### 11.2 局限

- 后端所有逻辑集中在一个文件，后续维护会变重
- 没有真正数据库，也没有事务和复杂并发控制
- 鉴权非常轻量，只适合本地/教学场景
- 题型配置定义没有在前后端之间做共享包复用
- 统计逻辑只对部分题型做了专门支持
- 代码里有一些编码痕迹和风格不完全统一的问题

---

## 12. 如果你后续要重构，优先考虑什么

优先级从高到低建议如下：

1. 把 `wenjuan-server/index.js` 拆成 `routes + services + db`
2. 把问卷组件 schema 抽成共享类型
3. 统一管理端列表页的数据加载模式
4. 给后端补更明确的参数校验
5. 给填写端补客户端校验和更完整的错误反馈

---

## 13. 最短学习路径

如果你现在时间很少，只做这 3 件事：

## 14. 一句话总结

这是一个典型的“配置驱动问卷系统”练习项目。
